Product Requirement Document (PRD)
Project Name

DevForge CLI — Modular Boilerplate Ecosystem for Modern Web Apps

1. Product Overview
Vision

DevForge CLI is a modular boilerplate generator that enables developers to instantly scaffold production-ready applications with a clean architecture, optimized performance, and pluggable features.

It aims to eliminate the repetitive work developers face when starting projects such as:

setting up architecture

configuring authentication

managing environment variables

structuring services

writing error handling

setting up database layers

Instead, developers can run:

npx devforge-cli

and generate a ready-to-run, production-grade codebase.

20. Boilerplate & Plugin Extensibility Architecture
Purpose

To ensure DevForge CLI can grow into a full ecosystem, the architecture must allow:

easy creation of new boilerplates

independent plugin development

safe integration between plugins and templates

minimal maintenance overhead

The system must follow a modular extensibility architecture.

20.1 Template Definition Standard

Every boilerplate template must follow a template definition contract.

Each template contains a configuration file:

template.config.ts

Example:

export default {
  name: "saas",
  description: "Multi-tenant SaaS application",
  stack: {
    runtime: "bun",
    backend: "hono",
    frontend: "solidjs",
    database: "postgres"
  },
  supportedPlugins: [
    "auth",
    "payment",
    "email",
    "analytics"
  ]
}

This allows the CLI to:

validate compatibility

prevent unsupported plugin installs

generate proper scaffolding

20.2 Plugin Architecture

Plugins are designed as self-contained modules.

Each plugin includes:

plugin.config.ts
plugin.install.ts
plugin.files/

Example structure:

plugins
 ├ auth
 │  ├ plugin.config.ts
 │  ├ plugin.install.ts
 │  └ files
 │     ├ routes
 │     ├ services
 │     └ middleware

Example plugin configuration:

export default {
  name: "auth",
  description: "Authentication module",
  compatibleWith: [
    "saas",
    "cms",
    "marketplace"
  ],
  dependencies: [
    "database"
  ]
}
20.3 Plugin Installation Lifecycle

Plugin installation follows this lifecycle.

Step 1 — validate compatibility

template.supportedPlugins

Step 2 — check dependency graph

Example:

auth -> database
payments -> auth

Step 3 — install plugin files

Files are copied into appropriate modules.

Step 4 — run plugin installer

Example:

plugin.install.ts

Used for:

updating configs

registering routes

updating env variables

20.4 Dependency Graph Validator

To prevent architectural issues, DevForge CLI implements a dependency graph validator.

Example graph:

AuthService
  -> UserRepository
  -> EmailService

If circular dependency occurs:

AuthService -> UserRepository -> AuthService

CLI will show:

Error: circular dependency detected

This protects long-term maintainability.

20.5 Auto Permission Binding

Permissions should automatically attach to routes.

Instead of manually defining permissions:

Example route file:

routes/products.ts
export const permissions = {
  create: "product.create",
  update: "product.update"
}

CLI generates middleware automatically:

requirePermission("product.create")

Benefits:

consistent security

easier maintenance

20.6 Migration Conflict Resolver

When multiple plugins modify database schema, migrations can conflict.

DevForge CLI introduces an Auto Migration Merger.

Example scenario:

Branch A migration

add users table

Branch B migration

add payments table

CLI merges automatically:

migration_merged.sql

Benefits:

fewer git conflicts

safer CI pipelines

20.7 Template Composition Engine

Templates are built from components instead of static files.

Example structure:

template

SaaS template =
